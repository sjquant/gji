# Commands

## Everyday commands

| Command | What it does |
| --- | --- |
| `gji new [branch] [--detached] [--json]` | Create a branch and worktree, then enter it. |
| `gji pr <ref> [--json]` | Fetch a pull request ref and open it in a dedicated worktree. |
| `gji go [branch] [--print]` | Jump to a worktree or print its path for scripting. |
| `gji root [--print]` | Jump back to the main repo root. |
| `gji status [--json]` | Show repo health, active worktrees, and ahead/behind counts. |
| `gji ls [--json]` | Show a compact worktree list. |
| `gji sync [--all]` | Fetch and rebase worktrees onto the default branch. |
| `gji clean [--force] [--json]` | Interactively prune stale worktrees. |
| `gji remove [branch] [--force] [--json]` | Remove a worktree and its branch. |
| `gji trigger-hook <hook>` | Re-run a configured hook in the current worktree. |
| `gji config [get\|set\|unset] [key] [value]` | Read and update configuration values. |
| `gji init [shell]` | Print or install shell integration. |

## Command groups

### Creation and entry

- `gji new`
- `gji pr`
- `gji go`
- `gji root`

These are the commands you use for opening or switching context.

### Inspection

- `gji status`
- `gji ls`

Use these when you need to understand the current set of worktrees and their state.

### Maintenance

- `gji sync`
- `gji clean`
- `gji remove`
- `gji trigger-hook`

These are the commands that keep the worktree set healthy over time.

### Configuration and shell setup

- `gji config`
- `gji init`

Use these to shape default behavior and shell handoff.

## Common patterns

### Start feature work

```bash
gji new feature/auth-refactor
```

### Open a pull request for review

```bash
gji pr 1234
```

### Re-run your setup hook

```bash
gji trigger-hook afterCreate
```

### Script against printed paths

```bash
repo_root=$(gji root --print)
feature_path=$(gji go --print feature/auth-refactor)
```

## JSON output

Several commands support `--json` so shell scripts and tools can consume structured output directly:

- `gji new --json`
- `gji pr --json`
- `gji status --json`
- `gji ls --json`
- `gji clean --json`
- `gji remove --json`
- `gji sync --json`

## Headless mode and GitHub Actions

Set `GJI_NO_TUI=1` when `gji` runs in CI, scheduled jobs, or any environment where a prompt would block the process:

```bash
GJI_NO_TUI=1 gji new --json automation/example
GJI_NO_TUI=1 gji remove --json --force automation/example
```

`GJI_NO_TUI=1` disables interactive prompts and fails fast when a command needs a missing branch or confirmation. Commands that remove worktrees still require `--force`; `--json` gives automation a stable response body on success or error.

The official GitHub Actions example is [`gji-headless-example.yml`](https://github.com/sjquant/gji/blob/main/.github/workflows/gji-headless-example.yml). It is a manual `workflow_dispatch` workflow for CI and automation jobs that need a temporary worktree.

Required inputs and secrets:

- No custom secrets are required.
- `actions/checkout` uses the default `GITHUB_TOKEN` with `contents: read`.
- The optional `branch` input names the temporary worktree branch. If omitted, the workflow uses `automation/gji-headless-<run id>`.

Expected behavior:

- The workflow checks out the repository with full history so Git worktrees can be created normally.
- It installs `@solaqua/gji@latest`, sets `GJI_NO_TUI=1`, and runs `gji status --json`.
- It creates a temporary worktree with `gji new --json`, stores the returned `path`, and runs automation from that directory.
- The cleanup step runs even after failures and removes the temporary worktree with `gji remove --json --force`.
